## 2) Общие обязательные требования (для обоих треков)

### 2.1. Tool-based агент, не “промпт в чате”

Ваш LLM-слой должен работать как **оркестратор tools**:

- у tools должны быть **понятные интерфейсы** (схемы входов/выходов, ошибки, ретраи, идемпотентность);

### 2.2. Автозапуск

Должен быть **автоматический запуск**:

- либо по **триггеру** (например, смена статуса задачи в Linear),
- либо по **расписанию** (например, ежемесячная агрегация/отчет).

### 2.3. Доступ к контексту: код и/или документация

Агент должен уметь **сам** подтягивать контекст из GitHub:

- Markdown knowledge base,
- релевантный код (поиск/чтение файлов/диффов/модулей),
- без ручного “вставьте контекст сюда”.

### 2.4. Coding-agents / глубокий анализ кода (желательно)

Если возможно — используйте или интегрируйте подходы уровня coding-agents:

- анализ модулей, куда человек/команда контрибьютит,
- анализ всей кодовой базы (ограниченно, но осмысленно),
- либо статический анализ + LLM-интерпретация результатов.

### 2.5. Модели: конкретика и аргументация

Нужно явно указать:

- какие **конкретные модели или инструменты** вы используете.
- почему именно они (качество, скорость, цена, контекст, reasoning, code-понимание),
- где какая модель применяется (оркестрация, суммаризация, code review, retrieval, кластеризация сигналов).

### 2.6. UX для разработчика

Результат должен быть удобен в ежедневной работе:

- артефакты появляются там, где живет команда (Linear comment / GitHub PR / Markdown doc),
- понятный формат, минимальный шум, возможность быстро поправить.

# Трек A — Planning: AI-агент для груминга и архитектурного планирования

## A1) Сценарий

После product design задача/эпик переходит в статус вроде **“Ready for Eng / Needs Grooming”** в Linear.

Система автоматически:

1. забирает задачу из Linear (описание, критерии, макеты/ссылки),
2. находит релевантные Markdown-доки и участки кода в GitHub,
3. делает **драфт**:
    - груминга (декомпозиция, зависимости, риски, вопросы),
    - архитектурного плана (варианты, изменения, API/схемы, миграции, rollout),
4. публикует результат в Linear + (опционально) кладет Markdown-док в репо и открывает PR.

## A2) Acceptance criteria (как проверяем)

- Автозапуск реально работает (webhook или polling/schedule).
- Агент сам подтягивает контекст (не просит “вставьте код сюда”).
- Результат помогает разработчику: понятная декомпозиция + архитектурный план, а не “общие слова”.
- Есть механизм “человеческой правки” (например, PR, или секция TODO/questions).
- Показана роль coding-анализа (хотя бы базовая): что он считал и как это повлияло на план.


**Надёжность и стоимость моделей**
1. Model fallback chain — Opus → Sonnet при rate limit / outage. Агент продолжает работу с пометкой о деградации качества, а не падает
2. Dynamic model selection по сложности задачи — S-задачи генерят ADR через Sonnet (–$0.30/задача), L/XL — полноценный Opus + extended thinking
3. Token budget cap ($2/задача через env var) — `UsageTracker.assertBudget()` перед каждым LLM-шагом, предотвращает runaway costs
4. Драфт моделей для удешевления. Минимум Haiku 4.5 (DOR) → GPT-5.4 Mini. Плюс потестить Haiku **Thinking Mode** для декомпозиции. Ещё возможный драфт — Opus 4.5 → Gemini 2.5 Pro / OpenAI GPT-5
