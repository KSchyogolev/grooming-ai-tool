## Шаги разработки

1. POC на Perplexity Computer - проверка гипотезы и работы всех инструментов в связке
2. Прототип первой версии с телефона на Claude Code
3. Отладка на Cursor Desktop в Dry Run режиме
4. Прогон всего пайплайна c ручным запуском, отладка промптов и UX
5. Деплой на Vercel + настройка Webhook
6. Написание отчетов и запись видео

## Осознанные оверхеды (допускались в рамках тестового задания)

1. Избыток тестов - покрытие выше необходимого, тем более для MVP.
2. Архитектурный анализ и сообщения для пользователя содержат больше деталей чем, пожалуй, нужно - для более наглядной демонстрации работы тула.
3. Разделение read_file и analyze_file в разные тулы. Можно было при чтении файла сразу возвращать и контент, и метаданные. Тогда вместо двух вызовов (read + analyze) был бы один, а LLM получал бы всё сразу. Тут выбор по 2м причинам - более структурированные данные и более "сложный" пайплайн для демонстрации работы оркестратора.
4. Строгость DOR-gate для наглядности демонстрации, в реальных условиях будет много false-positive срабатываний, нужнен дотюн под конкретный проект и сложившиеся практики описанияя задач.
5. Вебхуки для ревалидации сейчас реагируют на любой реакт/комент в Linear для демонстрации. В будущем нужна доп валидация и мб ветвление сценария AI-grooming в зависимости от поведения пользователя.
6. Переизбыток колонок со статусами в Linear, также для более детальной демонстрации.


## Дальнейшие шаги по улучшению (MVP → Production)

### Технические

1. Динамический выбор моделей - S-задачи через Sonnet, L/XL - Opus + extended thinking. 
2. Смена моделей. Минимум Haiku 4.5 (DOR) → GPT-5.4 Mini для удешевления. Плюс потестить Haiku **Thinking Mode** для декомпозиции. Ещё возможный драфт - Opus 4.5 → Gemini 2.5 Pro / OpenAI GPT-5. Плюс менеджмент тарифов, сейчас низкий rate limit приводит к высокому latency.
3. Zod-схемы как единый источник истины. 
4. Контекст связанных задач из Linear. LLM видит полную картину эпика при декомпозиции.
5. Вместо наивных слайсов `.slice(0, 1500)` сохранять imports, сигнатуры и типы. LLM получает структуру модуля, а не обрезанное тело.
6. Self-evaluation. Оценка ADR через Haiku (или другую легковесную модель).
7. Сбор успешных анализов с diff от разработчиков → eval-бенчмарк для A/B тестирования промптов и моделей. Дотюн skills - веса, оценки и декомпозиция с учетом контекста и бестпрактис компании.
8. AST вместо heuristic complexity. Текущая реализация - regex + indent depth покрывает подавляющее большинство кейсов.
9. Очереди и идемпотентность по ним. Сейчас, если одной и той же таске менять статус (при условии что груминг еще не готов) - она будет перезапускаться повторно. Можно было бы сделать костыль через дополнительного сообщения в задачу, но испортило бы UX. В рамках MVP решено было оставить такое поведение.
10. *Возможно* RAG/embeddings вместо agentic search - дешевле за lookup, но требует vector DB, индексацию и maintenance. Текущая реализация даёт лучшее качество контекста, так что нужен доресеч.

### Продуктовые

#### UX-фичи
1. Автоматическое создание sub-task в Linear из декомпозиции.
2. Confidence индикатор.
3. Re-groom при обновлении описания. Если описание задачи изменилось после грумнига, предложить перезапуск через реакцию.
4. Slack/Telegram/Email notification - нотификация для команды после успешного грумнига со ссылками на Linear и PR

#### Анализ
7. Связывание с тестами - для каждого файла найти `.test.`/`.spec.` файлы, показать "Test coverage: 3/5 key files have tests"
8. Детекция миграций и schema files - если задача затрагивает Prisma/GraphQL/SQL схемы, автоматически добавить секцию "Migration Plan" в ADR

#### Инфраструктура
9. Health check
10. Dead letter - после 3 неудачных попыток обработки прекращать повторы, маркировать `[AI-GROOMING-FAILED]`

#### Расширение пайплайна
11. Подключение трекеров вроде Granola для сбора дополнительного контекста из встреч и запуска тригеров
12. Web UI с админкой — дашборд обработанных задач, статистика cost/quality, ручной re-groom
